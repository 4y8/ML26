\begin{abstract}
  Effect handlers are gaining in popularity and existing languages start
  supporting them. However, existing type-and-effect systems lack either
  backward compatibility or robustness.

  We overview our implementation of a toy ML-like language supporting effects
  trying to solve both of these shortcomings. To do so, it combines modal effect
  types and a new type inference algorithm for first-class polymorphism.
\end{abstract}
\maketitle
\section{Introduction}
\subsection{Existing type-and-effect systems}
To take effects into account in the type system, traditional languages such as
Koka~\cite{koka2014} annotate arrows with the effects the function can
perform. Thus, each function inherits, in its signature, the effects of its
callees. This becomes cumbersome when typing higher-order functions and is an
obstacle to backward compatibility. For instance, in Koka, the function
\textsf{map} has the following type, even though it does not perform effects
itself.
\begin{lstlisting}[style=koka]
  forall <a, b, e> . (list<a>, (a) -> e b) -> e list<b>
\end{lstlisting}
Syntactic sugar can simplify such signatures, as demonstrated by
Frank~\cite{frank2020}. But, relying on syntax makes the system fragile.

The language Effekt~\cite{effekt2020} answers this problem with a
\emph{capability} based system. However, to guarantee effect-safety, functions
become second-class objects, hindering backward compatibilty.

\subsection{Modal Effect Types}
\textsc{Met}~\cite{tang2025modal, tang2026modal} draws inspiration from
Multimodal Type Theory~\cite{mtt2020} and represents effects a computation can
perform with \emph{modalities}. For instance, the \textsf{map} function has the
following type, where the modality \texttt{\small[]} indicates that the function
does not perform any effect.
\begin{lstlisting}[style=metl]
  [](forall a b . (a -> b) -> list a -> list b)
\end{lstlisting}

As modalities require explicit boxing and unboxing, one needs an inference
algorithm to turn \textsc{Met} into a programming language. The
original paper~\cite{tang2025modal} provides such an algorithm. But it only
deals with modalities and requires explicit type application.

We give an overview of \textsc{Met} that shows the bits we want to infer.

All typing judgements take a row of effects as input called the \emph{ambient
effect context} which indicates the effects the current computation can
perform. For instance, the expression $\Do \textsf{yield } 1$ is well-typed in
the effect context $\textsf{gen int}$, where $\textsf{gen } [a] = \textsf{yield}
: a \Rightarrow \textsf{unit}$.  The brackets around the type parameter
\textsf{a} indicate that it has kind \textsf{Abs}: the kind of types whose
meaning is independent from the ambient effect context. To avoid leakage, the
types of inputs and outputs of operation have kind \textsf{Abs}.

Modalities change the ambient effect contexts: either by replacing it
by an independent one (absolute modalities: $[E]$), or by removing and adding
some effects (relative modalities: $\langle L|D\rangle$). The $\keyw{mod}_\mu$
and $\keyw{let mod}_\mu$ constructs respectively introduce and eliminate
modality $\mu$. For example, this expression performs explicit subeffecting of a
variable $x:[](\textsf{int} \to\textsf{int})$:
\[
\left(\keyw{let mod}_{[]} y = x \keyw{ in } \keyw{mod}_{[E]}y\right) : [E](\textsf{int} \to\textsf{int}).
\]

Handlers also modify the effect context of the expression they encompass, to add
the effects they handle. In an empty effect context, this expression is
well-typed:
\[
\keyw{handle } (\Do \textsf{yield } 1) \keyw{ with }\{\textsf{yield } k\; n
\Rightarrow n \} : \textsf{int}.
\]

\subsection{Frost type inference}
The usual way to type check an application with a bidirectionnal type system is
to, first, infer the type of the function and then use this type to check the
argument~\cite{dunfield2013complete, greedy2023, jiang2025bidirectional}. Fewer
type systems allow the information to flow from the argument to the
function~\cite{pierce2000local, xie2018let}.

Frost supports information flow in both ways. To do so, the type system relies
on skeletons. Skeletons are types haunted by \emph{ghosts} ($\Hole$) and
\emph{universal ghosts} ($\TopHole$). When type checking an application, the
rule infers a skeleton for the argument, use that skeleton to infer a type for
the function and check the argument against the function type's argument. The
simplified pseudo-rule below illustrates this approach.
\begin{mathpar}
  \inferrule*[Lab=\textsc{T-App}]
  {
    \\
  }
  {\\}
\end{mathpar}

As noted in previous work on \textsc{Met}, elimination and introduction of
modalities remind instantiation and generalisation in type
inference~\cite[Section 7]{tang2025modal}. Frost implements this parallel thanks
to universal ghosts that unify the treatment of modalities and quantifiers.

We will review a toy programming language which combines Frost and modal effect
types. Examples in the core language and our new languages will demonstrate the
capabilities of the latter. Then, other examples will show some limitations of
our implementation. More examples, as well as a test suite are available
\href{https://github.com/4y8/modal-effect-types/tree/frost}{online}.

We lack proofs of completeness and soudness of the system.

\section{Practical programming with Modal Effect Types}
\subsection{Our implementation}
The core language of our implementation is \textsc{Met} with scoped
rows~\cite{scopedrows2005} as an effect structure (i.e.\ our modalities will
have scoped rows of effects) and some additions such as parametric effects and
algebraic data types.

\aghilas{remove or rephrase}
\textsc{Met} has modality-annotated variable declarations in its contexts:
$x:_\mu A$, where $A$ can also have a modality. For simplicity's sake, the
source language has usual plain bindings, and unboxes all the modalities of
variables.
\subsection{Itering on a list}
The standard iter higher-order function has type $\forall a . []((a \to
\textsf{unit})\to \textsf{unit})$ in \textsc{Met}.
\[
\Lambda a^{\textsf{Any}}.\keyw{mod}_{[]}\lambda f^{a\to\textsf{unit}} l^{\textsf{list }a}.\keyw{match } l \keyw{ with }\textsf{Nil}\to () \mid \textsf{Cons}(hd, tl)\to
\keyw{let mod}^{\langle\rangle}\_ = f hd \keyw{ in }
\keyw{let mod}^{[]}i = \textsf{iter} \{ a \} \keyw{ in }
i\; f\; tl
\]

The bidirectionnal type system of \textsc{Metl} avoids explicit modality unboxing.
\begin{lstlisting}[style=metl]
  val iter : forall a . []((a -> unit) -> list a -> unit)
  let iter f l =
    match l with
    | Nil -> ()
    | Cons (hd, tl) -> f hd; iter @a f tl
    end
\end{lstlisting}
And Frost removes the type application.
\begin{lstlisting}[style=metl]
    | Cons (hd, tl) -> f hd; iter f tl
\end{lstlisting}
\subsection{Finding in a list with effects}
Calling the iterator from the last subsection with an effectful function as
argument can find an element in a list, thanks to the parametric effect below.
\begin{lstlisting}[style=metl]
  eff gen [a] = yield : a => unit
\end{lstlisting}
Once again, \textsc{Metl} eliminates the need of modality boxing and unboxing.
\begin{lstlisting}[style=metl]
  val find :
    forall [a] . []((a -> bool) -> list a -> option a)
  let find p l =
    handle<gen a> iter @a
      (fun x -> if mask<gen> (p x) then do yield x else ())
      l with
    | return _ => None
    | yield x k => Some (x)
    end
\end{lstlisting}
The brackets that follow the \lstinline[style=metl]{handle} keyword provide a
parameter for the effect \lstinline[style=metl]{gen}. Frost erases the need for
this annotation as well as the explicit type application.

\subsection{Some limitations}
Our current unification rules have a syntactical treatment of quantifiers. Thus
we can unify neither \lstinline[style=metl]{[l]<l'>(unit -> unit)} and
\lstinline[style=metl]{[l', l](unit -> unit)},
nor \lstinline[style=metl]{forall a b . a * b} and
\lstinline[style=metl]{forall b a . a * b}, while they are both subtype
of eachother in both cases.

The rule to infer lambda abstractions introduces a fresh type variable for the
argument. As the system lacks kind inference, the kind of this variable is
\textsf{Any} to be permissive. Therefore, functions that have to take pure
values need an annotation. For instance, the \lstinline[style=metl]{find}
function of the previous subsection does not type check without the top-level
annotation because the variable \lstinline[style=metl]{x} is not pure.
