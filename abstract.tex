\begin{abstract}
  % Effect handlers are a powerful programming construct allowing users
  % to define and combine computational effects.
  %
  Effect systems statically track the effects that a program can perform.
%
  Traditional effect systems rely on effect variables to provide modular type
  signatures for higher-order functions.
%
  Modal effect types use modalities to give modular and succinct types to
  higher-order functions, reducing the need for effect variables meanwhile still
  allowing higher-order functions to be instantiated with different effects.
%
  However, the practical implementation of modal effect types, and in particular
  how to infer modalities, remains largely unexplored.

  We overview our implementation of a toy ML-like programming language with
  modal effect types.
%
  Our main novelty is that we extend \Frost, a new type inference algorithm for
  first-class polymorphism, to infer first-class modalities.
\end{abstract}

\section{Introduction}
\subsection{Existing type-and-effect systems}
To take effects into account in the type system, traditional languages such as
Koka~\cite{koka2014} annotate arrows with the effects the function can
perform. Therefore, each function inherits, in its signature, the effects of its
callees. This becomes cumbersome when typing higher-order functions and is an
obstacle to backward compatibility. For instance, if we annotate arrows with the
effects they can perform between square brackets, the function \lstinline{map} has
the following type, even though it does not perform any effect itself.
\begin{lstlisting}[style=metl]
  forall e . (list a, a ->[e] b) ->[e] list b
\end{lstlisting}
The effect variable \lstinline{e} allows \lstinline{map} to take any effectful
function as argument. Syntactic sugar can simplify such signatures, as
demonstrated by Frank~\cite{frank2020}. But relying on syntax makes the system
fragile.

The language Effekt~\cite{effekt2020} solves this problem with a
\emph{capability} based system. However, to guarantee effect-safety, functions
become second-class objects, which hinders backward compatibility.

Modal effect types remove the need for effect variables by not tying
effects to arrows ~\cite{tang2025modal, tang2026modal}.
%
An \emph{effect context} denotes the effects that a computation can perform.
%
A \emph{modality} is a transformation on effect contexts.
%% , typically the
%% currently active effect context which we refer to as the \emph{ambient effect
%% context}.
%
%% %draws inspiration from Multimodal Type Theory~\cite{mtt2020} and
%% \emph{effect contexts} represent the effects a computation can perform;
%% \emph{modalities} then modify effect contexts.
%
The \Met core calculus implements these ideas. It is an extension of System F
that adds effect handlers and modalities.
%
Modalities may appear directly in types such as $\mu A$, where $\mu$ is a
modality and $A$ is a type. A value of type $\mu A$ is a value of type $A$ only
available by first applying the modality $\mu$ to the ambient effect
context. This transformation acts hereditarily on every arrow in $A$.
%
For instance, the \textsf{map} function has the following type, where $[]$ is a
modality that indicates that the function does not perform any effect.
\[
  \forall a b . []((a \to b) \to \textsf{list } a \to \textsf{list } b)
\]
In general, $[E]$ denotes an \emph{absolute} modality, which overwrites the
input effect context with $E$.
%
Replacing the ambient effect context by the empty one may seem overly
restrictive. However, when we instantiate \textsf{map} and eliminate the
modality we may soundly upcast the empty effect to whatever the ambient effect
context happens to be.
%

%% input effect context with $E$.
%% %

%
%% In this case $E$ is empty.

%
%% , which means that it replaces
%% the input effect context with a
%
%% the ambient
%% effect context with an independent one; here the empty context. In general, the
%% modality $[E]$ replaces the ambient effect context with $E$. In the example
%% above, the modality implies that if any of the arrow in the type of \textsf{map}
%% performs an effect, then this effect comes from the ambient context of one of
%% the parent computations.
%
\Met also provides another kind of modality: a \emph{relative} modality $\langle
L|D\rangle$ removes the effects in $L$ from the ambient effect context before
adding the effects in $D$. Relative modalities are important for typing effect
handlers.

\Met provides explicit constructs for introducing and eliminating modalities.
%
At the term level, the $\keyw{mod}_\mu$ and $\keyw{let mod}_\mu$ constructs
respectively introduce and eliminate modality $\mu$. For example, the following
expression performs explicit subeffecting of a variable $x:[](\textsf{int}
\to\textsf{int})$ in any effect context:
\[
\left(\keyw{let mod}_{[]} y = x \keyw{ in } \keyw{mod}_{[E]} y\right) : [E](\textsf{int} \to\textsf{int})
\]
The $\keyw{mod}_\mu V$ expression modifies the effect context of the value $V$
by applying modality $\mu$. Let-bindings allow us to retrieve the original
value: $\keyw{let mod}_\mu x = \keyw{mod}_\mu V \keyw{ in } x$ reduces to $V$.
%
See~\cite{tang2025modal, tang2026modal} for a complete introduction to modal
effect types.


Just as writing explicit type abstractions and type applications in programs is
inconvenient, writing explicit introduction and elimination forms for modalities
is too.
%
The original \Met paper~\cite{tang2025modal} provides a bidirectional algorithm
that translates a surface language called \Metl into \Met. To dinstiguish it
from the languages we will introduce later, we will refer to the original \Metl
as \MetlF. But this algorithm only infers modalities: \MetlF still requires
explicit type applications for value polymorphism.
%
A central aim of our work is to combine modality inference with a bidirectional
typing algorithm for first-class polymorphism.

\subsection{\Frost type inference}
The usual way to type check an application with a bidirectional type system is
to, first, infer the type of the function and then use this type to check the
argument~\cite{dunfield2013complete, greedy2023, jiang2025bidirectional}. Less
commonly, some type systems allow the information to flow from the argument to
the function~\cite{pierce2000local, xie2018let}.

\Frost supports information flow in both directions~\cite{tang2025ML}. To do so,
the type system relies on \emph{skeletons}. Skeletons are types haunted by
\emph{ghosts} ($\Hole$) and \emph{universal ghosts} ($\TopHole$),
i.e.\ skeletons are types with holes in type or quantifier position. When type
checking an application, the rule infers a skeleton for the argument, uses that
skeleton to infer a type for the function and checks the argument against the
function type's argument. The skeleton allows information flow from the argument
to the function before it flows in the other direction.

For instance, consider two variables $\textsf{poly'}:\forall a.(\forall b. b\to
b\times a)\to a$ and $\textsf{ids}:\textsf{list }(\forall a.a\to a)$. \Frost
infers the type $\textsf{list }(\forall a.a\to a)$ for the expression
$\textsf{poly'} (\lambda x. (x, \textsf{ids}))$.
%
First, the skeleton $\TopHole.\Hole\to\Hole\times(\TopHole.\textsf{list
}(\forall a.a\to a))$ of the argument instantiates the function's type variable
$a$ to $\textsf{list }(\forall a.a\to a)$.
%
Then, checking the argument against $\forall b.b\to b\times\textsf{list
}(\forall a.a\to a)$ wraps the argument in a type abstraction.

As noted in previous work on \Met, elimination and introduction of modalities is
reminiscent of instantiation and generalisation in type inference~\cite[Section
  7]{tang2025modal}. Following this analogy, universal ghosts allows \Frost to
treat quantifiers and modalities with similar rules.

We will review a toy programming language which extends \Frost to infer modal
effect types, which we call \MetlFrost. We will use examples in \Met, \MetlF and
\MetlFrost to demonstrate the capabilities of the latter. Then, we will show
some limitations of \MetlFrost. More examples, as well as a test suite are
available \href{https://github.com/4y8/modal-effect-types/tree/frost}{online}.

\section{Practical programming with Modal Effect Types}
\subsection{Our implementation}
The core language of our implementation is \Met with scoped
rows~\cite{scopedrows2005} as an effect structure (i.e.\ modalities will have
scoped rows of effects) with additions such as parametric effects and algebraic
data types.

We have implemented three surface languages for this extension of \Met, which
all support modality inference: a version with explicit type application, one
with type inference for types without modalities or quantifiers based on an
algorithm of Dunfield and Krishnaswami~\cite{dunfield2013complete}, and one with
type inference based on \Frost. They are all available
\href{https://github.com/4y8/modal-effect-types/}{online} and this last
implementation also has a
\href{https://www.normalesup.org/~boussaa/met/playground/}{web interface}. The
web page features several effectful examples. They all only need require
top-level type annotations to type check. This makes us believe that \MetlFrost
is a usable surface language for modal effect types.

The new subtyping and unification rules for modalities look like the ones for
forall quantifiers. But they are more involved as modalities hold more
information. For instance, unification of two modal types also unifies the two
top-level modalities.

\end{lstlisting}

\subsection{Searching a list with effects}
Calling the iterator from the last subsection with an effectful function as
argument can search a list, thanks to the parametric effect below.
\begin{lstlisting}[style=metl]
  eff gen [a] = yield : a => unit
\end{lstlisting}
The declaration above defines a parametric effect \lstinline[style=metl]{gen}
with a single operation \lstinline[style=metl]{yield}. The brackets around the
parameter \lstinline[style=metl]{a} indicate that it has kind \textsf{Abs}: the
kind of types whose meaning is independent from the ambient effect
context. There is another kind, called \textsf{Any}, that includes all types. To
avoid effect leakage, the types of inputs and outputs of operations have kind
\textsf{Abs}.

Once again, \MetlF eliminates the need of modality introduction and elimination.
\begin{lstlisting}[style=metl]
  val find :
    forall [a] . []((a -> bool) -> list a -> option a)
  let find p l =
    handle<gen a> iter @a
      (fun x -> if mask<gen> (p x) then do yield x else ())
      l with
    | return _ => None
    | yield x k => Some (x)
    end
\end{lstlisting}
The angle brackets that follow the \lstinline[style=metl]{handle} keyword
provide a parameter for the effect \lstinline[style=metl]{gen}. \Frost erases the
need for this annotation, as well as for the explicit type application.

\subsection{Some limitations}
\paragraph{Syntactic treatment of quantifiers}
Our current unification rules have a syntactical treatment of quantifiers.  We
can unify neither $[l]\langle l'\rangle(\textsf{unit}\to\textsf{unit})$ and
$[l', l](\textsf{unit}\to\textsf{unit})$, nor $\forall a b . a \times b$ and
$\forall b a . a \times b$, while they are both subtype of each other in both
cases.

\paragraph{Lack of kind inference}
The type inference rule for lambda abstractions introduces a fresh type variable
for the argument. As the system lacks kind inference, the kind of this variable
is \textsf{Any} to be permissive. Therefore, functions that have to take pure
values need an annotation. For instance, the \lstinline[style=metl]{find}
function of the previous subsection does not type check without the top-level
annotation because the variable \lstinline[style=metl]{x} is not pure.

\subsection{Conclusion and future work}
The extension of \Frost to modal effect types is a promising approach to
integrate effect handlers in functional programming languages.

Future work includes taking effect contexts into account in skeleton inference,
extending \Frost to support a fragment of System $F^\omega$, proving the
soundness of the \Frost extension, and exploring a more semantic treatment of
quantifiers and modalities.
