\begin{abstract}
  Effect handlers are gaining in popularity and existing languages start
  supporting them. However, existing type-and-effect systems lack either
  backward compatibility or robustness. Modal effect types aim at solving both
  issues but lack type inference for values. To be user-friendly, modal effect
  types also need automatic boxing and unboxing of modalities.

  We overview our implementation of a toy ML-like programming language that
  combines modal effect types and Frost, a new type inference algorithm for
  first-class polymorphism. This algorithm deals with modalities and quantifiers
  in the same way, which allows type and modalities inference.
\end{abstract}

\section{Introduction}
\subsection{Existing type-and-effect systems}
To take effects into account in the type system, traditional languages such as
Koka~\cite{koka2014} annotate arrows with the effects the function can
perform. Therefore, each function inherits, in its signature, the effects of its
callees. This becomes cumbersome when typing higher-order functions and is an
obstacle to backward compatibility. For instance, if we annotate arrows with the
effects they can perform between square brackets, the function \lstinline{map} has
the following type, even though it does not perform any effect itself.
\begin{lstlisting}[style=metl]
  forall e . (list a, a ->[e] b) ->[e] list b
\end{lstlisting}
The effect variable \lstinline{e} allows \lstinline{map} to take any effectful
function as argument. Syntactic sugar can simplify such signatures, as
demonstrated by Frank~\cite{frank2020}. But relying on syntax makes the system
fragile.

The language Effekt~\cite{effekt2020} solves this problem with a
\emph{capability} based system. However, to guarantee effect-safety, functions
become second-class objects, hindering backward compatibilty.

\subsection{Modal Effect Types}
Modal effect types (\Met) remove the need for effect variables by not tiing
effects to arrows ~\cite{tang2025modal, tang2026modal}. In \Met,
%draws inspiration from Multimodal Type Theory~\cite{mtt2020} and
\emph{effect contexts} represent the effects a computation; \emph{modalities}
then modify effect contexts. For instance, the \textsf{map} function has the
following type, where the modality $[]$ indicates that the function
does not perform any effect.
\[
  [](\forall a b . (a \to b) \to \textsf{list } a \to \textsf{list } b)
\]

As modalities require explicit boxing and unboxing, \Met needs an inference
algorithm to become a programming language. The original
paper~\cite{tang2025modal} provides such an algorithm that translates a surface
language called \Metl to \Met. But this algorithm only deals with modalities:
\Metl requires explicit type application for value polymorphism.

We now give an overview of \Met that highlights the bits we want to
infer. See~\cite{tang2025modal, tang2026modal} for a complete introduction.

All typing judgements take a row of effects as input called the \emph{ambient
effect context} which indicates the effects the current computation can
perform. For instance, the expression $\Do \textsf{yield } 1$ is well-typed in
the effect context $\textsf{gen int}$, where $\textsf{gen } [a] = \textsf{yield}
: a \Rightarrow \textsf{unit}$.  The brackets around the type parameter
\textsf{a} indicate that it has kind \textsf{Abs}: the kind of types whose
meaning is independent from the ambient effect context. To avoid leakage, the
types of inputs and outputs of operation have kind \textsf{Abs}.

Modalities change the ambient effect contexts: either by replacing it by an
independent one (absolute modalities: $[E]$), or by removing and adding some
effects (relative modalities: $\langle L|D\rangle$). In the type of
\textsf{map}, the absolute modality $[]$ replaces the effect context with an
empty one, which ensures that the function does not perform any effect
itself. The $\keyw{mod}_\mu$ and $\keyw{let mod}_\mu$ constructs respectively
introduce and eliminate modality $\mu$. For example, this expression performs
explicit subeffecting of a variable $x:[](\textsf{int} \to\textsf{int})$:
\[
\left(\keyw{let mod}_{[]} y = x \keyw{ in } \keyw{mod}_{[E]}y\right) : [E](\textsf{int} \to\textsf{int})
\]

Handlers also modify the effect context of the expression they encompass, to add
the effects they handle. In an empty effect context, this expression is
well-typed:
\[
\keyw{handle } (\Do \textsf{yield } 1) \keyw{ with }\{\textsf{yield } k\; n
\Rightarrow n \} : \textsf{int}
\]
but needs an annotation to specify the parameter of the effect \textsf{gen}.

\subsection{Frost type inference}
The usual way to type check an application with a bidirectionnal type system is
to, first, infer the type of the function and then use this type to check the
argument~\cite{dunfield2013complete, greedy2023, jiang2025bidirectional}. Fewer
type systems allow the information to flow from the argument to the
function~\cite{pierce2000local, xie2018let}.

Frost supports information flow in both ways. To do so, the type system relies
on \emph{skeletons}. Skeletons are types haunted by \emph{ghosts} ($\Hole$) and
\emph{universal ghosts} ($\TopHole$), i.e.\ skeletons are types with holes in
type or quantifier position. When type checking an application, the rule infers
a skeleton for the argument, use that skeleton to infer a type for the function
and check the argument against the function type's argument.

For instance, consider two variables $\textsf{poly'}:\forall a.(\forall b. b\to
b\times a)\to a$ and $\textsf{ids}:\textsf{list }\forall a.a\to a$. Frost infers
the type $\textsf{list }\forall a.a\to a$ for $\textsf{poly'} (\lambda x. (x,
\textsf{ids}))$. First, the skeleton
$\TopHole.\Hole\to\Hole\times\TopHole.\textsf{list }\forall a.a\to a$ of the
argument instantiate $a$ to $\textsf{list }\forall a.a\to a$. Then, the type
argument $\forall b.b\to b\times\textsf{list }\forall a.a\to a$ wraps the
argument in a type abstraction.

As noted in previous work on \Met, elimination and introduction of
modalities remind instantiation and generalisation in type
inference~\cite[Section 7]{tang2025modal}. Frost implements this parallel thanks
to universal ghosts that unify the treatment of modalities and quantifiers.

We will review a toy programming language which adapts Frost to infer modal
effect types, which we call \MetlFrost. Examples in \Met, \Metl and \MetlFrost
will demonstrate the capabilities of the latter. Then, other examples will show
some limitations of our implementation. More examples, as well as a test suite
are available
\href{https://github.com/4y8/modal-effect-types/tree/frost}{online}.

We lack proofs of completeness and soudness for the extension of Frost to modal
effect types.

\subsection{Our implementation}
The core language of our implementation is \Met with scoped
rows~\cite{scopedrows2005} as an effect structure (i.e.\ our modalities will
have scoped rows of effects) with some additions such as parametric effects and
algebraic data types.

\subsection{Finding in a list with effects}
Calling the iterator from the last subsection with an effectful function as
argument can find an element in a list, thanks to the parametric effect below.
\begin{lstlisting}[style=metl]
  eff gen [a] = yield : a => unit
\end{lstlisting}
Once again, \Metl eliminates the need of modality boxing and unboxing.
\begin{lstlisting}[style=metl]
  val find :
    forall [a] . []((a -> bool) -> list a -> option a)
  let find p l =
    handle<gen a> iter @a
      (fun x -> if mask<gen> (p x) then do yield x else ())
      l with
    | return _ => None
    | yield x k => Some (x)
    end
\end{lstlisting}
The brackets that follow the \lstinline[style=metl]{handle} keyword provide a
parameter for the effect \lstinline[style=metl]{gen}. Frost erases the need for
this annotation as well as the explicit type application.

\subsection{Some limitations}
\paragraph{Syntactic treatment of quantifiers}
Our current unification rules have a syntactical treatment of quantifiers.  We
can unify neither $[l]\langle l'\rangle(\textsf{unit}\to\textsf{unit})$ and
$[l', l](\textsf{unit}\to\textsf{unit})$, nor $\forall a b . a \times b$ and
$\forall b a . a \times b$, while they are both subtype of eachother in both
cases.

\paragraph{Lack of kind inference}
The rule to infer lambda abstractions introduces a fresh type variable for the
argument. As the system lacks kind inference, the kind of this variable is
\textsf{Any} to be permissive. Therefore, functions that have to take pure
values need an annotation. For instance, the \lstinline[style=metl]{find}
function of the previous subsection does not type check without the top-level
annotation because the variable \lstinline[style=metl]{x} is not pure.

\subsection{Conclusion and future work}
The extension of Frost to modal effect types is a promising approach to
integrate effect handlers in functional programming languages.

Future work includes: taking effect contexts into account in skeleton inference;
extending Frost to support a fragment of system $F^\omega$; and exploring a more
semantic treatment of quantifiers and modalities.
